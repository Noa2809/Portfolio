# OWASP Juice Shop — Security Audit
OBJECTIF :
Identifier, exploiter et documenter
les failles de sécurité présentes
dans une application e-commerce
intentionnellement vulnérable.
CIBLE :
OWASP Juice Shop (localhost)
MÉTHODE :
Black-box → reconnaissance manuelle
→ exploitation → writeup Markdown
STATUS :
[ COMPLETED ] 58/100 challenges

OWASP Juice Shop est une application web volontairement truffée de failles de sécurité — conçue pour apprendre le pentest dans des conditions réalistes.

Notre mission : explorer chaque fonctionnalité (compte, login, panier, administration), identifier les vulnérabilités et les documenter avec méthodologie, risques et actions correctives.

⚠ DISCLAIMER

Projet réalisé dans un cadre éducatif sur un environnement isolé et autorisé. Aucune exploitation sur des systèmes réels.

ARCHITECTURE CIBLE

Structure du projet

Layer 01 — Frontend
Interface utilisateur

Application Angular exposant les fonctionnalités publiques et connectées : catalogue produits, panier, profil, historique de commandes.

Layer 02 — Backend
API REST Node.js

Serveur Express gérant l'authentification JWT, les routes API, les requêtes BDD. Point d'entrée principal des attaques de type injection et contournement.

Layer 03 — Base de données
SQLite embarqué

Base de données locale contenant utilisateurs, commandes, produits. Vulnérable aux injections SQL — exploitation possible depuis les champs de recherche et login.

Layer 04 — Workflow
Flux utilisateur audité

Compte, login, panier, checkout, espace admin — chaque étape du parcours utilisateur contient des failles délibérément introduites et documentées.

Structure des writeups

Chaque challenge résolu donne lieu à un fichier Markdown structuré suivant la même méthodologie reproductible.

🗺️
Méthodologie

Description pas-à-pas de la démarche d'exploitation — reconnaissance, identification du vecteur, payload utilisé.

🔍
Vulnérabilité

Nature précise de la faille, classification OWASP, composant affecté et conditions d'exploitation.

⚠️
Risques

Évaluation de l'impact : confidentialité, intégrité, disponibilité. Scénario d'attaque réel associé.

🛡️
Actions correctives

Recommandations concrètes pour corriger la vulnérabilité côté développement et configuration.
